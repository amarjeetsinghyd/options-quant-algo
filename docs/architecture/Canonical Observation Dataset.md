# Institutional Permanent Memory: Canonical Observation Dataset v3.1 (Frozen)

This document is the final, frozen specification for the permanent observational memory architecture of the Scientific Research Institution. It serves as the definitive engineering reference for all future implementation.

---

## 1. Immutable Design Principles

### The Immutable Prime Directive
**Only archive deterministic observations. Never archive mathematical interpretations.**
*   **Archive:** Prices, Volumes, Bid, Ask, Bid Quantity, Ask Quantity, Open Interest, Raw Exchange Fields, Infrastructure Measurements.
*   **Do Not Archive:** VFI, VWAP, EMA, PCR, Greeks, Gamma Exposure, IV Calculations, ML Features, Strategy Outputs.
*Scientific Justification:* Derived metrics belong to the Interpretation Layer. They depend on specific mathematical libraries, risk-free interest rate assumptions, or transient strategy weights. They must always be dynamically regenerated by future models based purely on the deterministic observations.

### Replay Compatibility
**The Canonical Observation Dataset must be completely independent of the current BrainService implementation.**
Future replay engines, trading simulators, and ML feature builders must be able to regenerate indicators, features, and entirely new research pipelines using only the **Raw Tick Dataset** and this **Canonical Observation Dataset**, without executing any of today's production code. This guarantees the dataset remains a pure institutional scientific asset rather than a rigid software artifact.

### Immutable Versioning Policy
**Scientific institutions preserve history. They never rewrite history.**
*   Schema versions are absolutely immutable.
*   Existing historical datasets are **never** rewritten or retroactively altered to fit a new schema.
*   Any future schema evolution creates Version 3.2, 4.0, etc.
*   Historical observations remain permanently valid and queryable under the schema version that originally created them.

## 2. Scientific Provenance & Dataset Linkage

To ensure absolute traceability across decades of research, every Canonical Observation must carry cryptographic provenance and raw data lineage.

### Scientific Provenance Metadata
*   `schema_version` (String): e.g., "v3.1"
*   `snapshot_version` (String): e.g., "v1.0"
*   `collector_version` (String): e.g., "v5.2"
*   `feed_version` (String): e.g., "AngelOne_SmartConnect_v3"
*   `exchange_name` (String): e.g., "NSE/NFO"
*   `broker_name` (String): e.g., "AngelOne"
*   `timezone` (String): e.g., "Asia/Kolkata"
*   `observation_interval` (Integer): e.g., 60 (seconds)
*   `serialization_version` (String): e.g., "pandas_2.1_pyarrow_14.0"

### Observation Identity
Every Canonical Observation carries a globally unique identity, allowing future experiments, trades, and research papers to cite the exact same mathematical state.
*   `observation_uuid` (String): Globally unique identifier for this specific 1-minute state.
*   `observation_sequence` (Integer): Monotonically increasing ID for the trading day (e.g., 1, 2, 3... 375).
*   `market_session_id` (String): Unique identifier linking all observations belonging to the same continuous market day.

### Dataset Linkage (Lineage)
A researcher must be able to trace a 1-minute Canonical Observation back to the exact raw market ticks that created it.
*   `raw_partition_id` (String): Pointer to the specific Parquet file in the `raw_ticks` directory.
*   `raw_tick_start_offset` (Integer): The exact row index or byte offset in the raw partition where this minute began.
*   `raw_tick_end_offset` (Integer): The exact row index or byte offset in the raw partition where this minute ended.

## 3. The Observational Schema

### Infrastructure Reality
*   `local_observation_timestamp` (Datetime): Exact OS-clock millisecond the snapshot was taken.
*   `exchange_timestamp` (Datetime): Exchange-reported event time.
*   `latency_ms` (Float): Network and processing delay.
*   `connection_quality` (Enum): Categorization of packet loss or stalling.
*   `session_type` (Enum): [LIVE, PREOPEN, MUHURAT].

### Underlying Index
*   `index_ltp` (Float): Absolute last traded price.
*   `index_canonical_ohlc` (Struct): The machine's locally aggregated OHLC.

### Index Futures
*   `futures_ltp` (Float) & `futures_volume` (Integer): Raw pricing and traded quantity.
*   *(Note: Premium/Discount is excluded as it is a mathematical interpretation [Spot - Futures]. Future models will derive this from `index_ltp` and `futures_ltp`)*

### Dynamic Option Chain
*The schema makes zero hard-coded assumptions about strike distance. It archives the exact chain available from the live feed, whether that is 100 strikes today or 1,000 strikes tomorrow.*
*   `strike_price` (Float) & `option_type` (CE/PE)
*   `last_traded_price` (Float) & `last_traded_quantity` (Integer)
*   `best_bid_price` (Float) & `best_ask_price` (Float)
*   `best_bid_qty` (Integer) & `best_ask_qty` (Integer)
*   `open_interest` (Integer): Absolute open contracts. *(OI Change is excluded as derived)*
*   `volume_for_the_day` (Integer)

### Constituent Stocks
*   `constituent_ltp` (Float) & `constituent_vtt` (Integer): The raw price and cumulative volume for every stock in the index.

### Cross-Market Context
*   `india_vix_ltp` (Float): Raw exchange volatility ticker.
*   `advancing_issues_count` (Integer) & `declining_issues_count` (Integer): Direct observation of constituent tick direction.

## 4. Final Data Lake Layout

The physical storage architecture is a deeply partitioned, ZSTD-compressed Parquet Data Lake, structured to separate raw physics from metadata and integrity layers.

```text
institutional_memory/
│
├── raw_ticks/                      # The asynchronous, raw event stream ground truth
│   ├── underlying/
│   ├── futures/
│   ├── options/
│   └── constituents/
│       └── YYYY/MM/DD/ticks.parquet
│
├── canonical_observations/         # The synchronized 1-minute state snapshots
│   ├── index_state/
│   ├── option_chain_state/
│   └── constituent_state/
│       └── YYYY/MM/DD/state.parquet
│
├── metadata/                       # Scientific Provenance
│   ├── schemas/
│   ├── feed_versions/
│   └── collector_versions/
│
├── manifests/                      # Integrity Verification (1 per partition/day)
│   └── YYYY/MM/DD/manifest.json
│
└── replay_logs/                    # Logs to ensure complete reproducibility of future experiments
```

## 5. Dataset Integrity Layer

To guarantee that a Scientific Institution can prove data has suffered zero corruption or silent degradation over decades, every daily partition will automatically generate a cryptographic integrity manifest.

**Example `manifest.json`:**
```json
{
  "date": "2026-06-27",
  "row_count": 1425000,
  "symbol_count": 256,
  "start_timestamp": "2026-06-27T09:15:00.000Z",
  "end_timestamp": "2026-06-27T15:30:00.000Z",
  "missing_intervals": ["11:22:00", "11:23:00"],
  "checksum_sha256": "8f434346648f6b96df89dda901c5176b10a6d83961dd3c1ac88b59b2dc327aa4",
  "compression": "ZSTD",
  "schema_version": "v3.1"
}
```

By enforcing this specification, the repository guarantees that its observational memory is immortal, immutable, mathematically pure, and entirely agnostic to whatever specific trading strategy or AI model happens to be deployed at the time.
